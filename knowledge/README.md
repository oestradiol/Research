# Knowledge

## 1. What this package is

**Knowledge** is a public, Obsidian-first knowledge atlas founded on the **Structured Unity Framework (SUF)**.

It is designed to be:

- a navigable map of major knowledge areas and subareas
- a graph-friendly study package rather than a linear handbook
- a sibling package to SUF rather than a replacement for it
- broad enough to move beyond SUF when the subject matter demands it
- explicit about where SUF is genuinely useful as scaffold, route logic, and relation discipline

Current snapshot:

- `v1.0.0` dated `2026-04-02`
- 9 cluster gateways
- 70 node notes
- typed relation vocabulary
- interconnected study routes

This is the first bounded public release of the Knowledge package inside the umbrella repository. It is stable and navigable, but it is not the main academic claim of the repository.

## 2. How to use it

- See `../structured-unity-framework/README.md` for the sibling SUF package.

- Start with `_indexes/knowledge-index.md`.
- Use `_indexes/node-index.md` when you want the full program in one place.
- Use `map/` for cluster gateways and domain notes.
- Use `study-routes/` for guided traversals through the graph.
- Use `_relations/` to keep cross-links typed and legible.
- Read `suf-relationship.md` for the clearest statement of how this project uses SUF.

## 3. Relationship to SUF

SUF contributes three things especially well here:

- a layered scaffold for moving across knowledge clusters
- relation discipline for linking notes without flattening them
- route logic for tracing how foundational, formal, applied, and reflexive areas connect

But Knowledge is broader than SUF. Many notes should use domain-native frameworks, competing systems traditions, or non-SUF literatures more heavily than SUF itself. The point is not to pretend SUF explains everything; the point is to let SUF organize a coherent starting atlas where it helps, and then let better local tools lead where they should.

## 4. SUF role vocabulary

When a Knowledge note explicitly names SUF's role, the preferred categories are:

- `primary_scaffold`
- `supporting_scaffold`
- `domain_native_lead`

These correspond to the public phrases:

- primary scaffold
- supporting scaffold
- domain-native lead

Knowledge should not imply that SUF is the mandatory voice inside every node.

## 5. Working posture

Knowledge should stay:

- public and readable
- graph-native in Obsidian
- enthusiastic about SUF where SUF genuinely earns it
- honest about where SUF should recede behind better domain-native tools
- explicit when a note is SUF-led, SUF-supported, or mainly domain-native
